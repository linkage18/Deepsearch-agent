# DeepSearch Agents 项目经历 STAR 讲法

---

## 一、STAR 核心叙事

### Situation（背景）
我需要一个能深度检索和理解学术论文的智能助手。现有工具要么只做关键词匹配（Google Scholar），要么是黑盒（Perplexity 无法定制检索范围），无法在自有论文库 + 全网搜索 + 结构化数据库三个维度上做融合检索与深度分析。

### Task（任务）
基于 DeepAgents 框架构建一个多智能体协作系统，能够：
- 理解用户用自然语言描述的研究需求
- 同时检索本地论文库、学术搜索引擎和论文元数据库
- 对检索结果进行证据级引用校验
- 生成带引用的结构化综述报告

### Action（行动）
- **架构选型**：选择 DeepAgents EventStream 模式，而非 LangChain AgentExecutor（详见下节决策解释）
- **多智能体协作**：设计"一主三从"架构——主智能体（Orchestrator）负责任务分解与报告生成，三个子智能体分别负责网络搜索、数据库查询、论文库研读
- **检索流水线**：搭建四层检索（Vector → BM25 → RRF → MiniLM 重排序），融合语义相似度与关键词匹配
- **存储升级**：从原版的 JSON 文件存储升级到 SQLite WAL + MySQL 双库架构，解决并发写入冲突
- **流式反馈**：实现 WebSocket 实时事件推送，用户能看到 Agent 每一步的思考与调用
- **自托管部署**：Docker Compose 编排 4 个容器（backend + frontend + MySQL + SearXNG），Nginx 反向代理

### Result（成果）
- 基于 90 篇模拟论文 + 34 条人工标注 ground-truth 评测集，全链路 MRR 达到 **0.97**（混合 BM25+RRF 策略），相比纯向量检索（MRR 0.88）提升 **~10%**
- 系统支持三种 embedding 模式（mock/local/openai），可无痛切换
- 引用校验模块可自动检测生成的 claim 是否有证据支撑，标记 unfounded 引用
- 同步生成带引用的 Markdown 综述报告，支持论文卡片结构化展示
- 工具调用审计日志 + WebSocket 实时推送，实现全链路可观测

## 二、关键技术决策（面试官想听的不是"用了什么"，而是"为什么选"）

### 1. 为什么不用 LangChain AgentExecutor？

| 对比维度 | LangChain AgentExecutor | DeepAgents EventStream |
|---------|------------------------|----------------------|
| 执行透明度 | 内部做了太多隐式处理（输出解析、重试、回退），出问题无法精确定位 | 每个事件都是结构化日志，每一步决策可见 |
| 控制粒度 | Agent Loop 是黑盒，难以插入自定义逻辑 | 完整控制 StateGraph 节点，可自由插入校验/中断/回调 |
| 子智能体 | 依赖 AgentExecutor 嵌套，调试困难 | 原生支持 interrupt-based 子智能体调用，隔离性更好 |

**面试话术**："我评估过 LangChain 的 AgentExecutor，它的问题在于内部做了太多'聪明'的事情——自动重试、自动解析输出、自动回退。在学术检索这种对精度要求高的场景里，这些隐式行为会让问题排查变得非常困难——你不知道到底哪一步出了错。DeepAgents 的 EventStream 模式让 Agent 的每一步决策都以事件的形式暴露出来，我可以精确追踪每个 tool_call 的输入输出，这对调试和审计来说至关重要。"

### 2. 为什么选 SearXNG 自托管搜索而非 Tavily？

- **成本**：Tavily 按 API 调用计费，SearXNG 自托管一次部署永久免费
- **可控性**：SearXNG 可自定义搜索引擎源（Google Scholar、 arXiv、Semantic Scholar 等），Tavily 只能用它预设的源
- **隐私**：用户搜索 query 不经过第三方 API，数据零外泄
- **调用链路**：从 `SearxngSearch` → `SearxSearchTool` → Agent 原生集成，切换成本低

**面试话术**："原版用的是 Tavily API，我换成了 SearXNG 自托管。这不是简单的替换——我需要自己搭一个元搜索引擎容器，配置它聚合 arXiv、Google Scholar、Semantic Scholar 等多个学术搜索源。代价是运维复杂了一点（多了一个容器），但换来了完全免费的搜索能力、对数据源的完全控制、以及用户搜索 query 不外泄的隐私保障。对于学术研究场景，这些优势比 Tavily 的便利性重要得多。"

### 3. 为什么用 SQLite WAL 而非 JSON 文件存储？

- **并发安全**：JSON 文件在多用户并发写时存在竞态条件，SQLite 通过 WAL 模式支持并行读 + 串行写
- **查询能力**：SQL 可以按时间戳、thread_id、事件类型做结构化过滤，JSON 只能全量加载后过滤
- **一致性**：`BEGIN IMMEDIATE` 事务防止 `SQLITE_BUSY`，写操作原子性有保障

**面试话术**："原版用的是 JSON 文件存储会话数据。但在多用户场景下很快暴露了问题——两个用户同时提交任务时，JSON 文件写入会互相覆盖。升级到 SQLite WAL 解决了这个问题：WAL 模式允许多个读事务和单个写事务同时进行，`BEGIN IMMEDIATE` 确保写操作不会因并发而失败。迁移代价也很低——SQLite 是文件型数据库，不需要单独部署一个数据库服务。"

### 4. 为什么设计四层检索流水线而非一步到位？

| 层级 | 技术 | 解决的问题 | 候选集规模 |
|------|------|-----------|-----------|
| 第一层 | 向量检索（LlamaIndex） | 语义召回，找到主题相关的文档 | 全库 → top-K |
| 第二层 | BM25 检索 | 关键词精确匹配，补充向量漏掉的精确术语 | top-K → top-K |
| 第三层 | RRF 融合 | 无参数融合两个互补排序，消除单算法偏差 | top-K → top-N |
| 第四层 | MiniLM 重排序 | 精排，在候选集上做更细粒度的语义匹配 | top-N → final |

**面试话术**："很多人问为什么不用 end-to-end 的检索模型一步到位。原因是每一层解决的是不同的问题：向量检索擅长'意思相近但用词不同'的语义召回，BM25 擅长精确的关键词命中。把 BM25 跑在全库上太慢了，所以先用向量做一个快速的 recall 剪枝（候选集），再在候选集上做 BM25 精排——这本质上是一个级联剪枝策略。RRF 的作用是消除单一排序算法的偏差，最后 MiniLM 做微调排名。每一层的代价都是递增的，但倒过来看，每一层处理的候选集规模都在递减，整体效率最优。"

### 5. 为什么用 WebSocket 而非轮询？

- **实时性**：Agent 每一步决策（tool_call 调用、思考内容、子智能体反馈）秒级推送
- **节省带宽**：只有有事件时才推送，不用定期轮询空接口
- **可观测性**：WebSocket 帧天然支持结构化 JSON 事件，每个事件带类型 + 时间戳，前端可精确渲染进度

**面试话术**："我需要让用户看到 Agent 在做什么——不是最终给一个报告就完了，而是实时看到 Agent 在搜索论文、在分析证据、在写作。HTTP 轮询要么延迟高，要么浪费带宽。WebSocket 是双向通道，前端连接一次就能持续接收 Agent 执行中的每个事件。更重要的是，我设计了断线重连机制——用户刷新页面后通过 `/api/tasks/{id}/events` 拉取历史事件，状态完全恢复。"

### 6. 为什么用 ContextVar 而非 threading.local？

- **异步安全**：`threading.local` 在 `asyncio` 中不工作——多个协程在同一个线程中切换，但 `threading.local` 绑定的线程不变，数据会"串台"
- **隐式传播**：`ContextVar` 会随着 `await` 调用自动传播到子协程，子协程拾取调用链上的上下文

**面试话术**："这是一个很容易踩的坑。`asyncio` 是单线程并发，多个协程共享一个线程——所以 `threading.local` 不会区分它们。`ContextVar` 是 Python 3.7 引入的方案，它绑定的是'上下文'而非线程。每个协程有一个独立的上下文，`await` 时会自动传播。我用它来隔离每个用户会话的工作目录，保证多个用户同时提交任务时各自的文件不会串到别人的目录里。"

### 7. 为什么用 BEGIN IMMEDIATE 而非普通 BEGIN？

- **SQLITE_BUSY 预防**：普通 `BEGIN` 是延迟事务——只在第一条写语句执行时才尝试获取写锁。如果此时另一个写事务正在进行，会返回 `SQLITE_BUSY`（5 秒后超时）
- **WAL 模式下的行为优化**：`BEGIN IMMEDIATE` 立即获取写锁，如果冲突则立刻等待，而不是在写了一部分后才发现冲突

**面试话术**："在 WAL 模式下，SQLite 的默认 `BEGIN` 是 DEFERRED 的——事务开始时没拿锁，直到第一条写操作才尝试拿。在高并发写入场景（比如多个 Agent 同时写事件日志），DEFERRED 事务容易在写了半截数据后撞上 `SQLITE_BUSY`，此时只能回滚重试。`BEGIN IMMEDIATE` 一进事务就声明'我要写'，SQLite 会立即尝试获取写锁——要么成功，要么阻塞——绝不会出现写到一半才发现冲突的情况。"

### 8. 为什么容器用容器名（`mysql:3306`）而非 `localhost`？

- **Docker DNS**：Docker Compose 创建了一个桥接网络，每个容器自动注册容器名为 DNS 记录
- **解耦**：容器之间通过服务名引用，即使物理端口映射变了（宿主机的 `3307:3306`），代码也不需要修改

**面试话术**："`localhost` 是宿主机层面的地址。两个容器各自有自己的网络栈，每个容器的 `localhost` 指向自己，不是对方。Docker Compose 会自动创建一个网络并为每个容器注册 DNS——所以 backend 容器里写 `mysql:3306` 就能解析到 mysql 容器的 IP。这也是微服务架构的一个基本模式：服务发现不是靠写死 IP，而是靠 DNS。"

## 三、Eval：量化能力的加分项

面试中能说出量化指标的候选人非常少。以下是本项目的关键数据：

### 检索系统评测结果

基于 90 篇模拟论文 + 34 条 ground-truth query（8 个主题簇，含简单/对比/精确数值三类难度）：

| 策略 | Recall@3 | Recall@5 | Recall@10 | MRR |
|------|----------|----------|-----------|-----|
| 纯向量检索（baseline） | 0.8775 | 0.9412 | 0.9902 | 0.8824 |
| 混合 BM25 + RRF | **0.9534** | **0.9681** | **0.9902** | **0.9706** |
| 全流水线（含 MiniLM 重排序） | 0.8309 | 0.9069 | 0.9902 | 0.8676 |

混合 BM25+RRF 相比纯向量检索：MRR 从 0.88 **提升至 0.97**（绝对提升 0.09），Recall@3 从 0.88 **提升至 0.95**（绝对提升 0.07）

> **注意**：全流水线含 MiniLM 重排序后 Recall/MRR 反而下降，原因是 cross-encoder 重排序将候选集从 20 压缩到 10，丢失了部分正确匹配的文档。这说明在本评测集中 BM25+RRF 组合已足够好，重排序更适合精准 Top-3 场景而非广度 Recall。

### 面试话术

"我建立了一个 34 条 query 的 ground-truth 评测集，覆盖 8 个 AI 主题簇（GNN、Transformer、RL、RAG、Few-shot、LLM Training、CV、Optimization），每条 query 设计了三种难度：单篇精确匹配、多篇对比、跨主题综合。在这个评测集上，我对比了三种检索策略：

- 纯向量检索（baseline）MRR 0.88，Recall@3 0.88
- 加上 BM25 + RRF 混合后 MRR 提升到 0.97，Recall@3 提升到 0.95
- 全链路加 MiniLM 重排序后 MRR 反而降到 0.87——重排序压缩候选集导致了召回损失

最关键的发现是：在 90 篇论文的中等规模论文库上，BM25 的 IDF 精确匹配能力对向量检索的补充效果非常显著——尤其是在跨主题对比 query 上。但 MiniLM 重排序在本场景中反而有害，因为它在压缩候选集的过程中丢掉了正确文档。这让我意识到：'加一层模型'不等于'效果更好'，必须用数据验证每个设计决策。"

## 四、面试话术补充

### 被问到"项目最大的挑战"

> "最大的挑战是检索召回率的优化。一开始纯向量检索在学术论文这种专门领域的效果很差——用户 query 中的术语向量化和 query 本身的语义不完全一致。比如'对比 Bayes 优化和进化优化的方法差异'——向量会把这两个概念映射到语义空间中相近的区域，但用户真正需要的是精确命中每种方法的核心论文。

> 我的解决思路是级联剪枝：先用向量检索做宽召回（candidate_multiplier=2），拉回更多候选集 → 再用 BM25 在候选集上做精确匹配 → RRF 融合两个排序 → MiniLM 重排序。迭代过程中我用 34 条人工标注 query 做评测，每次改动都跑一遍对比。

> 最大的一个教训是：一开始我把 BM25 的参数（k1, b）设成了默认值（1.5, 0.75），后来发现在学术论文这种文本上 k1 应该更低（经验值 1.2），b 应该更高（经验值 0.85），调整后在评测集上 Recall 有明显改善。"

### 被问到"如果重做，会改什么"

> "我会更早引入量化评测。前期我凭感觉调参——'嗯，这个结果看起来好了一些'。直到我设计了 34 条 ground-truth 评测集、跑了三种策略的 A/B 对比，才发现 BM25+RRF 才是真正拉动指标的组合（Recall@3 从 0.88 到 0.95），而 MiniLM 重排序实际上降低了效果（0.95 → 0.83）。如果更早搭建评测框架，可以少走很多弯路。另一个教训是论文库规模的重要性——最初 17 篇论文时所有策略 MRR 都在 0.95 以上，完全看不出差异。扩展到 90 篇后才真正拉开差距。所以 '有评测' 不等于 '有区分度的评测'。"

### 被问到"对比类似产品"

> "和 Perplexity 比，我们的优势是自托管——论文库可以私有化，搜索源可以自定义为学术专用源。和 ChatPDF 比，我们多了多论文交叉分析和引文校验。和传统的文献综述工具比，我们多了实时交互——用户可以看到 Agent 在检索什么、分析了什么证据，而不是黑盒出报告。"

## 五、简历描述模板

**科研文献证据管理与综述生成系统**

技术栈：Python、DeepAgents、LangGraph、LlamaIndex、BM25、FastAPI、WebSocket、SQLite、MySQL、SearXNG、React、Docker

项目背景：面向个人科研用户在文献调研分散、综述写作效率低的痛点，基于 DeepAgents 框架设计并实现多智能体 RAG 系统，支持论文上传、多源检索、证据级引用校验与综述初稿自动生成。

- **构建多源文献检索 Agent 流程与执行可观测机制**：基于 DeepAgents 与 LangGraph 设计 Orchestrator-Workers 架构（1 主 Agent + 3 子 Agent），拆分网络搜索、数据库查询、本地论文库检索三条工具链路；设计 WebSocket 实时事件推送与 SQLite WAL 持久化，Agent 每一步 tool_call 与决策事件即时推送至前端，支持断线重连后历史事件回放，提升多步骤任务的执行透明度与调试效率。

- **设计多策略论文证据检索与融合排序流水线**：基于 LlamaIndex 建立本地论文向量索引（chunk_size=512, overlap=64），叠加 BM25 关键词精确匹配与 RRF 无参数排序融合；构建 34 条 ground-truth 评测集（8 主题簇 × 3 难度等级）进行三维策略 A/B 对比，混合 BM25+RRF 策略 MRR 0.97（纯向量 0.88 → 混合 0.97，Recall@3 提升 0.08），返回含来源、页码和原文片段的结构化证据用于引用校验。

- **搭建证据结构化沉淀模块**：将检索证据依次整理为证据记录、论文卡片（含问题/方法/实验/结论/局限性）、横向对比矩阵和 Markdown 综述初稿，形成"检索 → 整理 → 对比 → 撰写"的材料整理闭环。

- **完善工程化质量保障与安全防护**：补充 API Key 鉴权、四层 SQL 注入纵深防护（前缀白名单 + 多语句拦截 + 关键字黑名单 + 表名白名单）、文件上传三层校验（类型/大小/数量）、ContextVar 会话隔离与任务超时保护（300s）；配套 255 项测试覆盖核心模块，支持三种 embedding 模式（mock/local/openai）与可插拔检索策略配置，降低部署与调试门槛。

## 七、常见追问深度准备

### 追问 1：为什么 MRR 只有 0.97？不应该更高吗？
> "0.97 是 BM25+RRF 混合策略的结果，已经很高了。纯向量是 0.88，融合后涨到 0.97。问题在于全链路（加 MiniLM 重排序）反而降到了 0.87——原因是 cross-encoder 重排序把候选集从 20 压缩到了 10，丢弃了部分正确文档。这说明在 90 篇论文的评测集上，BM25+RRF 的组合已经足够好，再加重排序反而有损召回广度。后续改进方向：1）扩大论文库到 500+ 篇，增加检索难度；2）优化 rerank 策略，只在 Top-10 精排而非压缩候选集；3）换更大的 embedding 模型（text-embedding-3-large）提升向量检索的 baseline。"

### 追问 2：如果论文量从几百篇增长到几十万篇，架构哪里会先崩？
> "三个瓶颈：第一，LlamaIndex 的本地向量索引是内存型加载的，几十万篇的 embedding 可能扛不住，需要外挂专业的向量数据库（Milvus/Qdrant）。第二，BM25 在过大的候选集上性能会急剧下降——现在的剪枝策略（先向量召回再 BM25 精排）就是为大规模设计的。第三，SQLite 不适合处理百万级的事件日志，届时需要迁移到 PostgreSQL 或专门的时序数据库。"

### 追问 3：你的引用校验是怎么做的？怎么保证不误判？
> "引用校验用 MiniLM 计算 claim 与 evidence 的余弦相似度，阈值设 0.25。低于 0.25 标记为 unfounded。这是一个保守策略——宁可漏标（false negative）也要防误判（false positive）。漏标的引用在最终报告中会被标记为'待人工确认'，而不是被悄悄删掉。代价是有些真实引用可能被标成 unfounded——如果 claim 改写程度太大或者抽取的 evidence 位置不对。后续改进方向是引入 LLM-as-a-Judge 做二次验证。"
