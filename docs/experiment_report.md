🔴 严重 MockEmbedding 导致向量检索实质无效

现象: 
  - 纯向量检索 Recall@3=0.075（10文档）vs 理论随机 ≈0.05
  - BM25+向量混合检索中 BM25 贡献有效排序，向量是噪声
  - 之前 3 篇论文时效果好是因为随机命中概率高（3/10≈0.3）
  - 10 篇文档时随机命中概率降为 0.075，问题暴露
根因:
  - .env 中 LLAMAINDEX_EMBED_MODEL=mock（默认配置）
  - MockEmbedding 生成 384 维随机向量，无语义信息
  - 之前所有实验报告的"向量检索"指标都是随机水平
  - 混合检索中 BM25 实际贡献了全部有效排序
影响:
  - 需要 LLAMAINDEX_EMBED_MODEL=openai 才能评估真实向量检索
  - 当前 BM25 是唯一有效的检索手段
  - 现有指标仅代表 BM25 + Reranker 的效果，不代表向量
修复方案:
  - .env 设置 LLAMAINDEX_EMBED_MODEL=openai
  - 或部署本地 embedding 模型（如 bge-small-zh）
  - 保留 BM25 作为兜底（对 Out-of-Distribution query 有效）

🟡 重要 中文 query + 英文文档的 BM25 不匹配

现象:
  - 中文 query 实验显示 jieba 分词后 BM25 几乎无法匹配英文文档
  - 结果几乎全部来自最大的 PDF（Li 2026, 19页），因为其 chunk 数最多
根因:
  - jieba 中文分词对英文文档的 BM25 分数 = 0（无重叠词）
  - MockEmbedding 随机向量下，chunk 数量多的文档被选中的概率更高
  - 文档的 chunk 数量严重倾斜（PDF 上百个 vs Markdown 1-3 个）
影响:
  - 中文 query + 英文文档 = 当前系统不支持跨语言检索
  - chunk 数分布需要均衡化
  - 建议为短文档做重叠 chunk 或复制策略

🟢 参考 RRF k 值在 BM25 主导时低敏感（k=5~60 效果一致）

实验数据:
  k=5:  R@3=0.275 MRR=0.275
  k=10: R@3=0.300 MRR=0.275
  k=30: R@3=0.275 MRR=0.275
  k=60: R@3=0.275 MRR=0.275
结论: 推荐 k=30 作为默认值。

🟢 参考 candidate_multiplier=2 足够，推荐降为 2

实验数据:
  mult=2: R@3=0.275 MRR=0.275
  mult=3: R@3=0.275 MRR=0.275
  mult=4: R@3=0.275 MRR=0.275
结论: mult=2 节省 33% 时间，效果相同。

🟢 参考 记忆系统 overlap_ratio 推荐 0.5（修正此前建议的 0.3）

实验数据:
  阈值 0.3: 精确率=1.00, 召回率=0.50, F1=0.67, 误报=0
  阈值 0.5: 精确率=1.00, 召回率=0.50, F1=0.67, 误报=0
  阈值 0.6: 精确率=1.00, 召回率=0.25, F1=0.40, 误报=0
  阈值 0.4 与 0.3 一致
结论: 0.3 和 0.5 的 F1 相同，但 0.5 更保守，推荐 0.5。
  之前建议 0.3 是因为测试集不同（3 篇论文的短标题）。

retrieval_config.py 推荐值:
  rrf_k: 30                 # k=5~60 无显著差异，取中间值确保迁移
  candidate_multiplier: 2   # mult=2 与 3 无差异，节省计算
  final_top_k: 5            # Recall 与延迟的平衡点
  bm25_tokenizer: null      # 英文场景 split；中文场景改 jieba
  rerank_model: all-MiniLM-L6-v2

  LLAMAINDEX_EMBED_MODEL=openai   ← 当前是 mock，需改为真实 embedding

  max_entries: 50
